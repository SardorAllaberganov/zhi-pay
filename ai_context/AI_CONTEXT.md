Save-as-draft secondary keeps its empty-content gate but also raises a toast on click when invalid. Pattern applies broadly — adopt for any future editor/form with similar "complete-this-before-submit" semantics |
| Notifications data model | **single-table extension** (Phase 19, 2026-05-03). Stayed with the canonical `notifications` table per `models.md §7` rather than introducing a separate `notification_sends`. Spec already locked `user_id NULL = broadcast/segment` and `deep_link jsonb {screen, params}` matching Stories' CTA shape — a second table would have contradicted that intent and forced cascading disambiguation across every cross-reference. Composer-side fields added in place: `status` enum (`scheduled / sending / sent / cancelled / failed`), `audience_type` enum (`broadcast / segment / single`), `audience_criteria` jsonb (segment-only), `scheduled_for`, `cancelled_at`, `cancellation_reason` text, `composed_by` FK admin_users, `recipient_count` snapshot, `delivered_count` / `opened_count` / `click_through_count` (nullable until sent). `is_read` / `read_at` clarified as per-user-only (audience_type=single). For mock-first dashboard work this models the admin-side composer cleanly; a real production system fans out to per-user inbox rows downstream — that's a backend implementation detail outside the dashboard's concern |
| Notification audience criteria | **5-dimensional, all optional** (Phase 19, 2026-05-03). Persisted shape under `notifications.audience_criteria` (jsonb) when `audience_type=segment`: `tiers` (array of tier_0/1/2, empty = any), `languages` (array of uz/ru/en, empty = any), `hasLinkedCard` (boolean | null), `hasCompletedTransfer` (boolean | null), `lastLogin` (`lt7d` / `lt30d` / `gt30d` / `never` | null). Estimation runs as a pure function `estimateAudience(criteria)` against `mockUsers.ts` — real backend would be a count query against an active-users index. Active-user requirement: only `users.status='active'` rows are eligible recipients regardless of other criteria. Recipient-language breakdown for the Sent-detail Stats card uses the same filter intersected with `users.preferred_language` |
| Notification compliance gate | **typed-confirm "I confirm" string, localized** (Phase 19, 2026-05-03). Compliance-typed sends bypass user notification preferences per spec — that warrants a higher-friction gate. The send-confirm dialog adds an input field that must contain the localized confirmation string ("I confirm" / "Подтверждаю" / "Tasdiqlayman") before the primary CTA enables. Comparison is trim + lowercase (so "i confirm" / "I CONFIRM" both pass) — fights typos without making the gate cosmetic. Large broadcasts (>5000 recipients) get a separate warning banner but no typed-confirm — only compliance type triggers the typed gate |
| Notification cancellability | **scheduled rows can cancel; sent rows are terminal** (Phase 19, 2026-05-03). State machine in `docs/mermaid_schemas/notification_send_state_machine.md` makes `sent / failed / cancelled` terminal with no transition out — push notifications cannot be retracted on user devices once dispatched. Cancellation requires ≥20-char reason note (App Versions / Stories precedent) and audit-log row with `from=scheduled, to=cancelled`. Sent-detail page shows the Cancel-scheduled action bar only when `status='scheduled'`; already-sent notifications have no admin actions exposed |
| Notification keyboard chord | **`g+i` reserved, cosmetic-only for now** (Phase 19, 2026-05-03; placeholder reserved in Phase 18 HISTORY). `g+i` reads "go inbox" mnemonically. The dashboard's chord prefix-handler is documentation-only at this point (HelpOverlay + CommandPalette) — no actual key-listener fires the navigation.
